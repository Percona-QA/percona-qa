## 3. Test-design highlights from today's work

The new tests follow the patterns the 2026-05-12 report recommended and add three new ones worth adopting going forward:

1. **State invariants on the failure path, not just `pytest.raises`.** Every negative test also asserts the catalog is *unchanged* afterwards. Examples:
   - Duplicate-add tests verify the **original** `options::text` is byte-identical after the second failed add (catches half-applied writes).
   - Scope-mismatch negatives verify the *other* scope's row was not collaterally mutated.
   - Missing-required-arg CLI negatives verify the cluster still starts cleanly afterwards (proves the failed call didn't half-write the offline state file).

2. **Column-name-agnostic listing helpers.** The TAP suite used a `provider_name` column that no longer exists; pg_tde 2.2.0 catalogs the row as `id, name, type, options`. Helpers query `SELECT name FROM pg_tde_list_all_*()` and the documented signature is captured inline (see `_list_global_provider_names` docstring) so future readers don't try to "restore" the TAP column name.

3. **Doc-aligned signature negatives.** Three of the new CLI negatives directly mirror the documented argument lists from
   [pg_tde_change_key_provider](https://docs.percona.com/pg-tde/command-line-tools/pg-tde-change-key-provider.html):
   - `vault-v2 <url> <mount_path> <token_path> [<ca_path>]` — `ca_path` is the only optional positional → test asserts missing `token_path` is rejected.
   - `kmip <host> <port> <cert_path> <key_path> [<ca_path>]` — same pattern → missing `key_path` rejected.
   - Provider types are exactly `file`, `vault-v2`, `kmip` (NOT legacy `vault`) → test asserts `vault` (sans `-v2`) is rejected.

4. **Lenient error-message matching.** No phrase pinning. The 2026-05-13 work surfaced two phrase changes between TAP and current build:
   - Delete-while-in-use: TAP expected `"Can't delete a provider which is currently in use"`, current build emits `"cannot delete provider which is currently in use"`.
   - List-providers column: TAP queried `provider_name`, current build has `name`.

   Both were caught early because the new tests match substrings (`"in use"`, `"does not exist"`) and use scope-agnostic helpers. The 2026-05-12 report's lenient-assertion guidance proved its value within a single review cycle.

5. **End-to-end keyring relocation** (`test_change_file_provider_while_in_use_keeps_data_readable`) — the canonical operator workflow: add file provider, create encrypted table, populate, `shutil.copy` keyfile to new path, run `pg_tde_change_database_key_provider_file`, then assert (a) data still readable in the *same* session, and (b) data still readable *after restart*. Without this, a regression where the change updates the catalog but doesn't rebind the active key would only surface on the next restart — long after the original keyfile was deleted.

### 5.2 Template-database coverage in detail (new `test_template_databases.py`)

Before today, `CREATE DATABASE ... TEMPLATE` with pg_tde was exercised only as a side-effect of `test_recovery.py::TestRelfilenodeReuse` (2 tests, scoped to relfilenode-reuse on a standby). The new `TestPgTdeTemplateDatabases` class (14 tests) covers the operator-facing lifecycle directly.

**Configuration finding pinned by this file:** for `CREATE DATABASE` from a template that contains encrypted objects to succeed, the server-wide default principal key must be registered via `pg_tde_set_default_key_using_global_key_provider(<key>, <provider>)`. Per-database bindings set with `pg_tde_set_key_using_global_key_provider` only apply to the database they were called in — they are not inherited by databases that don't yet exist. Without the default key the operation fails with the clear, actionable error `principal key not configured`. The helper `_register_default_global_key()` applies this missing setup step on top of the `tde_primary` fixture.

Documented contracts pinned here:

* The misconfiguration symptom (no default global key → `principal key not configured`) is the documented failure mode and stays a clear, actionable error rather than silently producing an unreadable clone.
* `STRATEGY = file_copy` against a template containing encrypted objects is rejected by pg_tde even with the default key set — `file_copy` is a page-byte copy across distinct per-database keys, which is unsafe. The error hint points at `STRATEGY = wal_log`.
* `STRATEGY = wal_log` against an encrypted template **succeeds** once the default global key is registered — that is the documented path for cloning encrypted templates.

| Test | What it pins down |
|---|---|
| `test_pg_tde_extension_installable_in_template1` | `CREATE EXTENSION pg_tde` works inside `template1` once the global provider + server key are configured |
| `test_new_db_from_template1_inherits_pg_tde_extension` | Default `CREATE DATABASE` (no explicit `TEMPLATE`) inherits the extension from `template1` |
| `test_create_database_without_default_key_rejected_with_principal_key_error` | **Positive negative test:** without `pg_tde_set_default_key_using_global_key_provider`, `CREATE DATABASE` from an encrypted template is rejected with `principal key not configured`; no half-created DB is left behind |
| `test_new_db_inherits_encrypted_tables_from_template1` | **End-to-end positive:** with the default key registered, an encrypted `tde_heap` table created in `template1` clones into the new DB, stays on `tde_heap`, reports encrypted, AND is readable end-to-end — successful decryption proves the inherited default key is in use |
| `test_create_database_with_encrypted_template_rejects_file_copy` | **Positive negative test:** even with the default key registered, `STRATEGY = file_copy` against an encrypted template is rejected with a helpful hint pointing at `STRATEGY = wal_log` |

## 8. Recommendations

Carry forward from 2026-05-12 plus three new ones:

1. **Catalog dumps from a live build are the source of truth for column / function signatures.** Today's session caught `provider_name` → `name` from a dump shared partway through (`tde_functions.log`). Test fixtures should reference the column metadata via `SELECT * FROM pg_tde_list_*()` + first-column extraction OR explicit column names confirmed against a current build — never copy-paste from TAP.

2. **State-invariant assertions on the failure path** (the negative tests added today). Adopt as a pattern: every `pytest.raises` should be paired with a check that the *other* observable state was untouched. Catches partial-state and side-effect regressions that pure exception-matching misses.

3. **Documented contracts → directly-mapped negative tests.** When a CLI / SQL surface has a published reference page, every documented constraint deserves at least one negative test. The 3 new doc-aligned CLI negatives (signature parameter counts, deprecated type names) follow this directly from [docs.percona.com/pg-tde](https://docs.percona.com/pg-tde/).
