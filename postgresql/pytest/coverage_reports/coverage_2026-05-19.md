### Headline

The minor-upgrade suite was refocused from a large checklist-style module (documented as 36 tests in the May-14 catalog) to **11 tests** plus a **two-phase staged workflow** driven by `--upgrade-data-dir`. This matches real operator practice: same PostgreSQL major, same `$PGDATA`, package swap, then `ALTER EXTENSION pg_tde UPDATE`.

A new module **`test_pdg_migration.py`** (10 tests) ports scenarios from [Percona PG 18 migration doc](https://docs.percona.com/postgresql/18/migration.html) (same-server stop/start, `pg_dumpall`, physical copy, `pg_basebackup`, pg_tde variants).

## 2. Staged minor upgrade — automation

### 2.1 Script: `postgresql/pytest/run_minor_upgrade_workflow.sh`

| Phase | Action |
|---|---|
| 1 | `setup_test_env.sh --install-pkgs --pg-major 18.3 --repo-component release` |
| 2 | Pytest Setup → writes `/var/lib/pg_tde_minor_upgrade/{single,ha}/` + `upgrade_state.json` |
| 3 | `setup_test_env.sh --install-pkgs --pg-major 18.4 --repo-component testing` |
| 4 | Pytest Verify → start saved PGDATA, `ALTER EXTENSION pg_tde UPDATE`, assert digests |

**Options:** `--with-pg2381`, `--setup-only`, `--upgrade-only`, `--verify-only`, `--skip-install`, `--upgrade-data-dir PATH`.

**Guards:** Verify refuses to run if installed Postgres version string does not contain `NEW_PG_MAJOR` (e.g. still 18.3 when target is 18.4).

### 2.2 Persistent layout (`PG_TDE_UPGRADE_DATA_DIR`)

```text
/var/lib/pg_tde_minor_upgrade/
├── single/          # WAL enc, 500-row tde_heap, global key
│   ├── pgdata/
│   ├── sock/
│   ├── keyfile.per
│   └── upgrade_state.json
├── single_pg2381/   # optional: PG-2381 churn (VACUUM FULL + drop/recreate)
└── ha/              # primary + standby copy, global key
    ├── nodeA/
    ├── nodeB/
    ├── sock/
    └── upgrade_state.json
```

### 2.3 Harness changes (`conftest.py`, `test_tde_minor_upgrade.py`)

| Change | Why |
|---|---|
| `pytest.mark.minor_upgrade` on staged Setup/Verify classes | Staged tests no longer skipped by `upgrade` marker requiring `--old-install-dir` |
| `minor_upgrade` collection skip without `--upgrade-data-dir` | Clear error vs wrong skip reason |
| `_capture_pre_upgrade_state` churn branch | PG-2381 setup does not create `staged_minor_upgrade` |
| `ReplicationManager.rewire_standby_conninfo()` | HA Verify: replica `primary_conninfo` pointed at ephemeral Setup port |
| Idempotent Verify (500 vs 600 rows) | Safe re-run after partial Verify |
| `TestPg2381MinorUpgradeSetup` separate class | PG-2381 not collected by default Setup class |

## 4. `test_pdg_migration.py` (10 tests) — **new**

**Marker:** `migration`, `slow`  
**Doc:** [Percona PG 18 migration](https://docs.percona.com/postgresql/18/migration.html)

| Class | Tests | Purpose |
|---|---|---|
| `TestMigrateOnSameServer` | 3 | Config backup/restore; same PGDATA stop/start; `pg_dumpall` → new cluster |
| `TestMigrateOnDifferentServer` | 3 | `pg_dumpall` restore; cold PGDATA copy; `pg_basebackup` seed |
| `TestMigrateWithPgTde` | 3 | TDE stop/start; `pg_dumpall` with TDE; config backup includes `pg_tde` |
| `TestMigratePgTdeCrossMinorVersion` | 1 | Same-datadir restart after pg_tde control bump (needs **same PG major** on both install dirs) |

**Harness notes:**

- `pg_dumpall` restore filters bootstrap `CREATE ROLE` for OS superuser (avoids "role already exists").
- Cross-minor test skips if `--old-install-dir` and `--install-dir` are different PG majors (e.g. 17 vs 18).

**Not in scope:** `apt`/`yum` package install (simulated via pytest clusters only).

## Appendix: May-14 catalog drift

The May-14 deep catalog **§11** described 36 tests across five checklist phases (`TestPostUpgradeState`, seven-test `TestAlterExtensionUpdate`, etc.). The current `test_tde_minor_upgrade.py` implements a **slimmer 11-test** layout focused on staged persistence and a smaller set of ephemeral HA checks. Treat **this report and §11 in the catalog (2026-05-19 refresh)** as authoritative for minor upgrade.
