10 // A Cache is an interface that maps keys to values. It has internal
11 // synchronization and may be safely accessed concurrently from
12 // multiple threads. It may automatically evict entries to make room
13 // for new entries. Values have a specified charge against the cache
14 // capacity. For example, a cache where the values are variable
15 // length strings, may use the length of the string as the charge for
16 // the string.
17 //
18 // A builtin cache implementation with a least-recently-used eviction
19 // policy is provided. Clients may use their own implementations if
20 // they want something more sophisticated (like scan-resistance, a
21 // custom eviction policy, variable cache sizing, etc.)

150 // A set of callbacks to allow objects in the primary block cache to be
151 // be persisted in a secondary cache. The purpose of the secondary cache
152 // is to support other ways of caching the object, such as persistent or
153 // compressed data, that may require the object to be parsed and transformed
154 // in some way. Since the primary cache holds C++ objects and the secondary
155 // cache may only hold flat data that doesn't need relocation, these
156 // callbacks need to be provided by the user of the block
157 // cache to do the conversion.
158 // The CacheItemHelper is passed to Insert() and Lookup(). It has pointers
159 // to callback functions for size, saving and deletion of the
160 // object. The callbacks are defined in C-style in order to make them
161 // stateless and not add to the cache metadata size.
162 // Saving multiple std::function objects will take up 32 bytes per
163 // function, even if its not bound to an object and does no capture.
164 //
165 // All the callbacks are C-style function pointers in order to simplify
166 // lifecycle management. Objects in the cache can outlive the parent DB,
167 // so anything required for these operations should be contained in the
168 // object itself.
169 //
170 // The SizeCallback takes a void* pointer to the object and returns the size
171 // of the persistable data. It can be used by the secondary cache to allocate
172 // memory if needed.
173 //
174 // RocksDB callbacks are NOT exception-safe. A callback completing with an
175 // exception can lead to undefined behavior in RocksDB, including data loss,
176 // unreported corruption, deadlocks, and more.
177 using SizeCallback = size_t (*)(void* obj);
