9 // A database can be configured with a custom FilterPolicy object.
10 // This object is responsible for creating a small filter from a set
11 // of keys. These filters are stored in rocksdb and are consulted
12 // automatically by rocksdb to decide whether or not to read some
13 // information from disk. In many cases, a filter can cut down the
14 // number of disk seeks form a handful to a single disk seek per
15 // DB::Get() call.
16 //
17 // Most people will want to use the builtin bloom filter support (see
18 // NewBloomFilterPolicy() below).

122 // Return a new filter policy that uses a bloom filter with approximately
123 // the specified number of bits per key. See
124 // https://github.com/facebook/rocksdb/wiki/RocksDB-Bloom-Filter
125 //
126 // bits_per_key: average bits allocated per key in bloom filter. A good
127 // choice is 9.9, which yields a filter with ~ 1% false positive rate.
128 // When format_version < 5, the value will be rounded to the nearest
129 // integer. Recommend using no more than three decimal digits after the
130 // decimal point, as in 6.667.
131 //
132 // To avoid configurations that are unlikely to produce good filtering
133 // value for the CPU overhead, bits_per_key < 0.5 is rounded down to 0.0
134 // which means "generate no filter", and 0.5 <= bits_per_key < 1.0 is
135 // rounded up to 1.0, for a 62% FP rate.
136 //
137 // The caller is responsible for eventually deleting the result, though
138 // this is typically handled automatically with BlockBasedTableOptions:
139 // table_options.filter_policy.reset(NewBloomFilterPolicy(...));
140 //
141 // As of RocksDB 7.0, the use_block_based_builder parameter is ignored.
142 // (The old, inefficient block-based filter is no longer accessible in
143 // the public API.)
144 //
145 // Note: if you are using a custom comparator that ignores some parts
146 // of the keys being compared, you must not use NewBloomFilterPolicy()
147 // and must provide your own FilterPolicy that also ignores the
148 // corresponding parts of the keys. For example, if the comparator
149 // ignores trailing spaces, it would be incorrect to use a
150 // FilterPolicy (like NewBloomFilterPolicy) that does not ignore
151 // trailing spaces in keys.
152 extern const FilterPolicy* NewBloomFilterPolicy(
153 double bits_per_key, bool IGNORED_use_block_based_builder = false);

155 // A new Bloom alternative that saves about 30% space compared to
156 // Bloom filters, with similar query times but roughly 3-4x CPU time
157 // and 3x temporary space usage during construction. For example, if
158 // you pass in 10 for bloom_equivalent_bits_per_key, you'll get the same
159 // 0.95% FP rate as Bloom filter but only using about 7 bits per key.
160 //
161 // The space savings of Ribbon filters makes sense for lower (higher
162 // numbered; larger; longer-lived) levels of LSM, whereas the speed of
163 // Bloom filters make sense for highest levels of LSM. Setting
164 // bloom_before_level allows for this design with Level and Universal
165 // compaction styles. For example, bloom_before_level=1 means that Bloom
166 // filters will be used in level 0, including flushes, and Ribbon
167 // filters elsewhere, including FIFO compaction and external SST files.
168 // For this option, memtable flushes are considered level -1 (so that
169 // flushes can be distinguished from intra-L0 compaction).
170 // bloom_before_level=0 (default) -> Generate Bloom filters only for
171 // flushes under Level and Universal compaction styles.
172 // bloom_before_level=-1 -> Always generate Ribbon filters (except in
173 // some extreme or exceptional cases).
174 //
175 // Ribbon filters are compatible with RocksDB >= 6.15.0. Earlier
176 // versions reading the data will behave as if no filter was used
177 // (degraded performance until compaction rebuilds filters). All
178 // built-in FilterPolicies (Bloom or Ribbon) are able to read other
179 // kinds of built-in filters.
180 //
181 // Note: the current Ribbon filter schema uses some extra resources
182 // when constructing very large filters. For example, for 100 million
183 // keys in a single filter (one SST file without partitioned filters),
184 // 3GB of temporary, untracked memory is used, vs. 1GB for Bloom.
185 // However, the savings in filter space from just ~60 open SST files
186 // makes up for the additional temporary memory use.
187 //
188 // Also consider using optimize_filters_for_memory to save filter
189 // memory.
190 extern const FilterPolicy* NewRibbonFilterPolicy(
191 double bloom_equivalent_bits_per_key, int bloom_before_level = 0);
