512 /**
513 Add all rows from a log file to the error-log ring-buffer.
514
515 We have to guesstimate where to start reading in the log:
516
517 - The error_log table is kept in a ring-buffer.
518 Reading more items than we have space for is therefore harmless;
519 we should however try to keep the waste down for performance
520 reasons.
521
522 - In the traditional log, the part of the row before the message
523 is 63 characters long. This gets converted into an event header.
524 The header's size is platform-dependent, but usually shorter
525 than 63 bytes. Thus, the size of each record in the input will
526 be more or less the size of its corresponding record in the
527 output. As a consequence, reading the ring-buffer's size from
528 the input should be about right.
529
530 - When reading the JSON log, we'll fill in the event header from
531 the parsed values, but we will also attach the entire JSON record
532 to the event. Each record in the ring-buffer is therefore the
533 size of the original JSON record, plus the size of a record header.
534 As a consequence reading the ring-buffer's size from the input
535 will give us more events than we need (because we "lose" about
536 50 bytes to the header for each event). However, the input is of
537 variable length and we can not tell whether it's a few long rows
