773 /*
774 Additional Comments
775 -------------------
776
777 This is for timing, i.e. finding out how long a piece of code
778 takes. If you want time of day matching a wall clock, the
779 my_timer_xxx functions won't help you.
780
781 The best timer is the one with highest frequency, lowest
782 overhead, and resolution=1. The my_timer_info() routine will tell
783 you at runtime which timer that is. Usually it will be
784 my_timer_cycles() but be aware that, although it's best,
785 it has possible flaws and dangers. Depending on platform:
786 - The frequency might change. We don't test for this. It
787 happens on laptops for power saving, and on blade servers
788 for avoiding overheating.
789 - The overhead that my_timer_init() returns is the minimum.
790 In fact it could be slightly greater because of caching or
791 because you call the routine by address, as recommended.
792 It could be hugely greater if there's an interrupt.
793 - The x86 cycle counter, RDTSC doesn't "serialize". That is,
794 if there is out-of-order execution, rdtsc might be processed
795 after an instruction that logically follows it.
796 (We could force serialization, but that would be slower.)
797 - It is possible to set a flag which renders RDTSC
798 inoperative. Somebody responsible for the kernel
799 of the operating system would have to make this
800 decision. For the platforms we've tested with, there's
801 no such problem.
802 - With a multi-processor arrangement, it's possible
803 to get the cycle count from one processor in
804 thread X, and the cycle count from another processor
805 in thread Y. They may not always be in synch.
806 - You can't depend on a cycle counter being available for
807 all platforms. On Alphas, the
808 cycle counter is only 32-bit, so it would overflow quickly,
809 so we don't bother with it. On platforms that we haven't
810 tested, there might be some if/endif combination that we
811 didn't expect, or some assembler routine that we didn't
812 supply.
813
814 The recommended way to use the timer routines is:
815 1. Somewhere near the beginning of the program, call
816 my_timer_init(). This should only be necessary once,
817 although you can call it again if you think that the
818 frequency has changed.
819 2. Determine the best timer based on frequency, resolution,
820 overhead -- all things that my_timer_init() returns.
821 Preserve the address of the timer and the my_timer_into
822 results in an easily-accessible place.
823 3. Instrument the code section that you're monitoring, thus:
824 time1= my_timer_xxx();
825 Instrumented code;
826 time2= my_timer_xxx();
827 elapsed_time= (time2 - time1) - overhead;
828 If the timer is always on, then overhead is always there,
829 so don't subtract it.
830 4. Save the elapsed time, or add it to a totaller.
831 5. When all timing processes are complete, transfer the
832 saved / totalled elapsed time to permanent storage.
833 Optionally you can convert cycles to microseconds at
834 this point. (Don't do so every time you calculate
835 elapsed_time! That would waste time and lose precision!)
836 For converting cycles to microseconds, use the frequency
837 that my_timer_init() returns. You'll also need to convert
838 if the my_timer_microseconds() function is the Windows
839 function QueryPerformanceCounter(), since that's sometimes
840 a counter with precision slightly better than microseconds.
841
842 Since we recommend calls by function pointer, we supply
843 no inline functions.
844
845 Some comments on the many candidate routines for timing ...
846
847 clock() -- We don't use because it would overflow frequently.
848
849 clock_gettime() -- In tests, clock_gettime often had
850 resolution = 1000.
851
852 gettimeofday() -- available on most platforms, though not
853 on Windows. There is a hardware timer (sometimes a Programmable
854 Interrupt Timer or "PIT") (sometimes a "HPET") used for
855 interrupt generation. When it interrupts (a "tick" or "jiffy",
856 typically 1 centisecond) it sets xtime. For gettimeofday, a
857 Linux kernel routine usually gets xtime and then gets rdtsc
858 to get elapsed nanoseconds since the last tick. On Red Hat
859 Enterprise Linux 3, there was once a bug which caused the
860 resolution to be 1000, i.e. one centisecond. We never check
861 for time-zone change.
862
863 getnstimeofday() -- something to watch for in future Linux
864
865 do_gettimeofday() -- exists on Linux but not for "userland"
866
867 get_cycles() -- a multi-platform function, worth watching
868 in future Linux versions. But we found platform-specific
869 functions which were better documented in operating-system
870 manuals. And get_cycles() can fail or return a useless
871 32-bit number. It might be available on some platforms,
872 such as arm, which we didn't test. Using
873 "include <linux/timex.h>" or "include <asm/timex.h>"
874 can lead to autoconf or compile errors, depending on system.
875
876 rdtsc, __rdtsc, rdtscll: available for x86 with Linux BSD,
877 Solaris, Windows. See "possible flaws and dangers" comments.
878
879 times(): what we use for ticks. Should just read the last
880 (xtime) tick count, therefore should be fast, but usually
881 isn't.
882
883 GetTickCount(): we use this for my_timer_ticks() on
884 Windows. Actually it really is a tick counter, so resolution
885 >= 10 milliseconds unless you have a very old Windows version.
886 With Windows 95 or 98 or ME, timeGetTime() has better resolution than
887 GetTickCount (1ms rather than 55ms). But with Windows NT or XP or 2000,
888 they're both getting from a variable in the Process Environment Block
889 (PEB), and the variable is set by the programmable interrupt timer, so
890 the resolution is the same (usually 10-15 milliseconds). Also timeGetTime
891 is slower on old machines:
892 http://www.doumo.jp/aon-java/jsp/postgretips/tips.jsp?tips=74.
893 Also timeGetTime requires linking winmm.lib,
894 Therefore we use GetTickCount.
895 It will overflow every 49 days because the return is 32-bit.
896 There is also a GetTickCount64 but it requires Vista or Windows Server 2008.
897 (As for GetSystemTimeAsFileTime, its precision is spurious, it
898 just reads the tick variable like the other functions do.
899 However, we don't expect it to overflow every 49 days, so we
900 will prefer it for my_timer_milliseconds().)
901
902 QueryPerformanceCounter() we use this for my_timer_microseconds()
903 on Windows. 1-PIT-tick (often 1/3-microsecond). Usually reads
904 the PIT so it's slow. On some Windows variants, uses RDTSC.
905
906 GetLocalTime() this is available on Windows but we don't use it.
907
908 getclock(): documented for Alpha, but not found during tests.
909
910 mach_absolute_time() and UpTime() are recommended for Apple.
911 Inititally they weren't tried, because asm_ppc seems to do the job.
912 But now we use mach_absolute_time for nanoseconds.
913
914 Any clock-based timer can be affected by NPT (ntpd program),
915 which means:
916 - full-second correction can occur for leap second
917 - tiny corrections can occur approimately every 11 minutes
918 (but they only affect the RTC which isn't the PIT).
919
920 We define "precision" as "frequency" and "high precision" as
921 "frequency better than 1 microsecond". We define "resolution"
922 as a synonym for "granularity". We define "accuracy" as
923 "closeness to the truth" as established by some authoritative
924 clock, but we can't measure accuracy.
925
926 Do not expect any of our timers to be monotonic; we
927 won't guarantee that they return constantly-increasing
928 unique numbers.
929
930 We tested with AIX, Solaris (x86 + Sparc), Linux (x86 +
931 Itanium), Windows, 64-bit Windows, QNX, FreeBSD, HPUX,
932 Irix, Mac. We didn't test with SCO.
933
