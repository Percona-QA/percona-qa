458 /** While scanning logs for multi-record mini-transaction (mtr), we have two
459 passes. In first pass, we check if all the logs of the mtr is present in
460 current recovery buffer or not. If yes, then in second pass we go through the
461 logs again the add to hash table for apply. To avoid parsing multiple times,
462 we save the parsed records in first pass and reuse them in second pass.
463
464 Parsing of redo log takes significant amount of time and this optimization of
465 avoiding second parse gave about 1.8x speed up on recovery scan time of 1G of
466 redo log from sysbench rw test.
467
468 There is currently no limit for maximum number of logs in an mtr. Practically,
469 from sysbench rw test recovery with 1G of redo log to recover from the record
470 count were spread from 3 - 1235 with majority between 600 - 700. So, it is
471 likely by saving 1k records we could avoid most of the re-parsing overhead.
472 Considering possible bigger number of records in other load and future changes
473 the limit for number of saved records is kept at 8k. The same value from the
474 contribution patch. The memory requirement 32 x 8k = 256k seems fine as one
475 time overhead for the entire instance. */
