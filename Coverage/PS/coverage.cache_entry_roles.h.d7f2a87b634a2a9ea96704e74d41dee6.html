54 // To associate cache entries with their role, we use a hack on the
55 // existing Cache interface. Because the deleter of an entry can authenticate
56 // the code origin of an entry, we can elaborate the choice of deleter to
57 // also encode role information, without inferring false role information
58 // from entries not choosing to encode a role.
59 //
60 // The rest of this file is for handling mappings between deleters and
61 // roles.
62
63 // To infer a role from a deleter, the deleter must be registered. This
64 // can be done "manually" with this function. This function is thread-safe,
65 // and the registration mappings go into private but static storage. (Note
66 // that DeleterFn is a function pointer, not std::function. Registrations
67 // should not be too many.)
68 void RegisterCacheDeleterRole(Cache::DeleterFn fn, CacheEntryRole role);
69
70 // Gets a copy of the registered deleter -> role mappings. This is the only
71 // function for reading the mappings made with RegisterCacheDeleterRole.
72 // Why only this interface for reading?
73 // * This function has to be thread safe, which could incur substantial
74 // overhead. We should not pay this overhead for every deleter look-up.
75 // * This is suitable for preparing for batch operations, like with
76 // CacheEntryStatsCollector.
77 // * The number of mappings should be sufficiently small (dozens).
