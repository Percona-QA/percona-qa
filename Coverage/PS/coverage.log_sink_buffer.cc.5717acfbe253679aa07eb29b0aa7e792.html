49 /**
50 Make sure only one instance of the buffered "writer" runs at a time.
51
52 In normal operation, the log-event will be created dynamically, then
53 it will be fed through the pipeline, and then it will be released.
54 Since the event is allocated in the caller, we can be sure it won't
55 go away wholesale during processing, and since the event is local to
56 the caller, no other thread will tangle with it. It is therefore safe
57 in those cases not to wrap a lock around the event.
58 (The log-pipeline will still grab a shared lock, THR_LOCK_log_stack,
59 to protect the pipeline (not the event) and the log-services cache from
60 being changed while the pipeline is being applied.
61 Likewise, log-services may protect their resources (file-writers will
62 usually take a lock to serialize their writes; the built-in filter will
63 take a lock on its rule-set as that is shared between concurrent
64 threads running the filter, and so on).
65 None of these are intended to protect the event itself though.
66
67 In buffered mode on the other hand, we copy each log-event (the
68 original of which, see above, is owned by the caller and local
69 to the thread, and therefore safe without locking) to a global
70 buffer / backlog. As this backlog can be added to by all threads,
71 it must be protected by a lock (once we have fully initialized
72 the subsystem with log_builtins_init() and support multi-threaded
73 mode anyway, as indicated by log_builtins_inited being true, see
74 below). This is that lock.
75 */
