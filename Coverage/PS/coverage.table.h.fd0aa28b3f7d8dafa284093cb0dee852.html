441 // This enum allows trading off increased index size for improved iterator
442 // seek performance in some situations, particularly when block cache is
443 // disabled (ReadOptions::fill_cache = false) and direct IO is
444 // enabled (DBOptions::use_direct_reads = true).
445 // The default mode is the best tradeoff for most use cases.
446 // This option only affects newly written tables.
447 //
448 // The index contains a key separating each pair of consecutive blocks.
449 // Let A be the highest key in one block, B the lowest key in the next block,
450 // and I the index entry separating these two blocks:
451 // [ ... A] I [B ...]
452 // I is allowed to be anywhere in [A, B).
453 // If an iterator is seeked to a key in (A, I], we'll unnecessarily read the
454 // first block, then immediately fall through to the second block.
455 // However, if I=A, this can't happen, and we'll read only the second block.
456 // In kNoShortening mode, we use I=A. In other modes, we use the shortest
457 // key in [A, B), which usually significantly reduces index size.
458 //
459 // There's a similar story for the last index entry, which is an upper bound
460 // of the highest key in the file. If it's shortened and therefore
461 // overestimated, iterator is likely to unnecessarily read the last data block
462 // from each file on each seek.
463 enum class IndexShorteningMode : char {
464 // Use full keys.
465 kNoShortening,
466 // Shorten index keys between blocks, but use full key for the last index
467 // key, which is the upper bound of the whole file.
468 kShortenSeparators,
469 // Shorten both keys between blocks and key after last block.
470 kShortenSeparatorsAndSuccessor,
471 };
472
473 IndexShorteningMode index_shortening =
474 IndexShorteningMode::kShortenSeparators;
