39 /**
40 @file mysys/lf_alloc-pin.cc
41 wait-free concurrent allocator based on pinning addresses.
42
43 It works as follows: every thread (strictly speaking - every CPU, but
44 it's too difficult to do) has a small array of pointers. They're called
45 "pins". Before using an object its address must be stored in this array
46 (pinned). When an object is no longer necessary its address must be
47 removed from this array (unpinned). When a thread wants to free() an
48 object it scans all pins of all threads to see if somebody has this
49 object pinned. If yes - the object is not freed (but stored in a
50 "purgatory"). To reduce the cost of a single free() pins are not scanned
51 on every free() but only added to (thread-local) purgatory. On every
52 LF_PURGATORY_SIZE free() purgatory is scanned and all unpinned objects
53 are freed.
54
55 Pins are used to solve ABA problem. To use pins one must obey
56 a pinning protocol:
57
58 1. Let's assume that PTR is a shared pointer to an object. Shared means
59 that any thread may modify it anytime to point to a different object
60 and free the old object. Later the freed object may be potentially
61 allocated by another thread. If we're unlucky that other thread may
62 set PTR to point to this object again. This is ABA problem.
63 2. Create a local pointer LOCAL_PTR.
64 3. Pin the PTR in a loop:
65 do
66 {
67 LOCAL_PTR= PTR;
68 pin(PTR, PIN_NUMBER);
69 } while (LOCAL_PTR != PTR)
70 4. It is guaranteed that after the loop has ended, LOCAL_PTR
71 points to an object (or NULL, if PTR may be NULL), that
72 will never be freed. It is not guaranteed though
73 that LOCAL_PTR == PTR (as PTR can change any time)
74 5. When done working with the object, remove the pin:
75 unpin(PIN_NUMBER)
76 6. When copying pins (as in the list traversing loop:
77 pin(CUR, 1);
78 while ()
79 {
80 do // standard
81 { // pinning
82 NEXT=CUR->next; // loop
83 pin(NEXT, 0); // see #3
84 } while (NEXT != CUR->next); // above
85 ...
86 ...
87 CUR=NEXT;
88 pin(CUR, 1); // copy pin[0] to pin[1]
89 }
90 which keeps CUR address constantly pinned), note than pins may be
91 copied only upwards (!!!), that is pin[N] to pin[M], M > N.
92 7. Don't keep the object pinned longer than necessary - the number of
93 pins you have is limited (and small), keeping an object pinned
94 prevents its reuse and cause unnecessary mallocs.
95
96 Explanations:
97
98 3. The loop is important. The following can occur:
99 thread1> LOCAL_PTR= PTR
100 thread2> free(PTR); PTR=0;
101 thread1> pin(PTR, PIN_NUMBER);
102 now thread1 cannot access LOCAL_PTR, even if it's pinned,
103 because it points to a freed memory. That is, it *must*
104 verify that it has indeed pinned PTR, the shared pointer.
105
106 6. When a thread wants to free some LOCAL_PTR, and it scans
107 all lists of pins to see whether it's pinned, it does it
108 upwards, from low pin numbers to high. Thus another thread
109 must copy an address from one pin to another in the same
110 direction - upwards, otherwise the scanning thread may
111 miss it.
112
113 Implementation details:
114
115 Pins are given away from a "pinbox". Pinbox is stack-based allocator.
116 It used dynarray for storing pins, new elements are allocated by dynarray
117 as necessary, old are pushed in the stack for reuse. ABA is solved by
118 versioning a pointer - because we use an array, a pointer to pins is 32 bit,
119 upper 32 bits are used for a version.
120 */
