56 /*
57 PartitionPruningModule
58
59 This part of the code does partition pruning. Partition pruning solves the
60 following problem: given a query over partitioned tables, find partitions
61 that we will not need to access (i.e. partitions that we can assume to be
62 empty) when executing the query.
63 The set of partitions to prune doesn't depend on which query execution
64 plan will be used to execute the query.
65
66 HOW IT WORKS
67
68 Partition pruning module makes use of RangeAnalysisModule. The following
69 examples show how the problem of partition pruning can be reduced to the
70 range analysis problem:
71
72 EXAMPLE 1
73 Consider a query:
74
75 SELECT * FROM t1 WHERE (t1.a < 5 OR t1.a = 10) AND t1.a > 3 AND t1.b='z'
76
77 where table t1 is partitioned using PARTITION BY RANGE(t1.a). An apparent
78 way to find the used (i.e. not pruned away) partitions is as follows:
79
80 1. analyze the WHERE clause and extract the list of intervals over t1.a
81 for the above query we will get this list: {(3 < t1.a < 5), (t1.a=10)}
82
83 2. for each interval I
84 {
85 find partitions that have non-empty intersection with I;
86 mark them as used;
87 }
88
89 EXAMPLE 2
90 Suppose the table is partitioned by HASH(part_func(t1.a, t1.b)). Then
91 we need to:
92
93 1. Analyze the WHERE clause and get a list of intervals over (t1.a, t1.b).
94 The list of intervals we'll obtain will look like this:
95 ((t1.a, t1.b) = (1,'foo')),
96 ((t1.a, t1.b) = (2,'bar')),
97 ((t1,a, t1.b) > (10,'zz'))
98
99 2. for each interval I
100 {
101 if (the interval has form "(t1.a, t1.b) = (const1, const2)" )
102 {
103 calculate HASH(part_func(t1.a, t1.b));
104 find which partition has records with this hash value and mark
105 it as used;
106 }
107 else
108 {
109 mark all partitions as used;
110 break;
111 }
112 }
113
114 For both examples the step #1 is exactly what RangeAnalysisModule could
115 be used to do, if it was provided with appropriate index description
116 (array of KEY_PART structures).
117 In example #1, we need to provide it with description of index(t1.a),
118 in example #2, we need to provide it with description of index(t1.a, t1.b).
119
120 These index descriptions are further called "partitioning index
121 descriptions". Note that it doesn't matter if such indexes really exist,
122 as range analysis module only uses the description.
123
124 Putting it all together, partitioning module works as follows:
125
126 prune_partitions() {
127 call create_partition_index_description();
128
129 call get_mm_tree(); // invoke the RangeAnalysisModule
130
131 // analyze the obtained interval list and get used partitions
132 call find_used_partitions();
133 }
134

601 /*
602 Collect partitioning ranges for the SEL_ARG tree and mark partitions as used
603
604 SYNOPSIS
605 find_used_partitions()
606 ppar Partition pruning context.
607 key_tree SEL_ARG range (sub)tree to perform pruning for
608
609 DESCRIPTION
610 This function
611 * recursively walks the SEL_ARG* tree collecting partitioning "intervals"
612 * finds the partitions one needs to use to get rows in these intervals
613 * marks these partitions as used.
614 The next session describes the process in greater detail.
615
616 IMPLEMENTATION
617 TYPES OF RESTRICTIONS THAT WE CAN OBTAIN PARTITIONS FOR
618 We can find out which [sub]partitions to use if we obtain restrictions on
619 [sub]partitioning fields in the following form:
620 1. "partition_field1=const1 AND ... AND partition_fieldN=constN"
621 1.1 Same as (1) but for subpartition fields
622
623 If partitioning supports interval analysis (i.e. partitioning is a
624 function of a single table field, and partition_info::
625 get_part_iter_for_interval != NULL), then we can also use condition in
626 this form:
627 2. "const1 <=? partition_field <=? const2"
628 2.1 Same as (2) but for subpartition_field
629
630 INFERRING THE RESTRICTIONS FROM SEL_ARG TREE
631
632 The below is an example of what SEL_ARG tree may represent:
633
634 (start)
635 | $
636 | Partitioning keyparts $ subpartitioning keyparts
637 | $
638 | ... ... $
639 | | | $
640 | +---------+ +---------+ $ +-----------+ +-----------+
641 \-| par1=c1 |--| par2=c2 |-----| subpar1=c3|--| subpar2=c5|
642 +---------+ +---------+ $ +-----------+ +-----------+
643 | $ | |
644 | $ | +-----------+
645 | $ | | subpar2=c6|
646 | $ | +-----------+
647 | $ |
648 | $ +-----------+ +-----------+
649 | $ | subpar1=c4|--| subpar2=c8|
650 | $ +-----------+ +-----------+
651 | $
652 | $
653 +---------+ $ +------------+ +------------+
654 | par1=c2 |------------------| subpar1=c10|--| subpar2=c12|
655 +---------+ $ +------------+ +------------+
656 | $
657 ... $
658
659 The up-down connections are connections via SEL_ARG::left and
660 SEL_ARG::right. A horizontal connection to the right is the
661 SEL_ARG::next_key_part connection.
662
663 find_used_partitions() traverses the entire tree via recursion on
664 * SEL_ARG::next_key_part (from left to right on the picture)
665 * SEL_ARG::left|right (up/down on the pic). Left-right recursion is
666 performed for each depth level.
667
668 Recursion descent on SEL_ARG::next_key_part is used to accumulate (in
669 ppar->arg_stack) constraints on partitioning and subpartitioning fields.
670 For the example in the above picture, one of stack states is:
671 in find_used_partitions(key_tree = "subpar2=c5") (***)
672 in find_used_partitions(key_tree = "subpar1=c3")
673 in find_used_partitions(key_tree = "par2=c2") (**)
674 in find_used_partitions(key_tree = "par1=c1")
675 in prune_partitions(...)
676 We apply partitioning limits as soon as possible, e.g. when we reach the
677 depth (**), we find which partition(s) correspond to "par1=c1 AND par2=c2",
678 and save them in ppar->part_iter.
679 When we reach the depth (***), we find which subpartition(s) correspond to
680 "subpar1=c3 AND subpar2=c5", and then mark appropriate subpartitions in
681 appropriate subpartitions as used.
682
683 It is possible that constraints on some partitioning fields are missing.
684 For the above example, consider this stack state:
685 in find_used_partitions(key_tree = "subpar2=c12") (***)
686 in find_used_partitions(key_tree = "subpar1=c10")
687 in find_used_partitions(key_tree = "par1=c2")
688 in prune_partitions(...)
689 Here we don't have constraints for all partitioning fields. Since we've
690 never set the ppar->part_iter to contain used set of partitions, we use
691 its default "all partitions" value. We get subpartition id for
692 "subpar1=c3 AND subpar2=c5", and mark that subpartition as used in every
693 partition.
694
695 The inverse is also possible: we may get constraints on partitioning
696 fields, but not constraints on subpartitioning fields. In that case,
697 calls to find_used_partitions() with depth below (**) will return -1,
698 and we will mark entire partition as used.
699
700 TODO

704 1 OK, one or more [sub]partitions are marked as used.
705 0 The passed condition doesn't match any partitions
706 -1 Couldn't infer any partition pruning "intervals" from the passed
707 SEL_ARG* tree (which means that all partitions should be marked as
708 used) Marking partitions as used is the responsibility of the caller.
709 */
