25 /**
26 * @defgroup MDC Metadata Cache
27 *
28 * Synopsis
29 * ========
30 *
31 * Metadata Cache plugin communicates with Metadata and Group Replication
32 * exposed by the cluster to obtain its topology and availablity information.
33 * The digest of this information is then exposed to Routing Plugin in form of a
34 * routing table.
35 * Key components:
36 * - Metadata Cache API - interface through which it exposes its service
37 * - Refresh Mechansim - responsible for updating routing table
38 *
39 *
40 *
41 *
42 *
43 * Glossary
44 * ========
45 *
46 * MD = metadata, several tables residing on the metadata server, which (among
47 * other things) contain cluster topology information. It reflects the
48 * desired "as it should be" version of topology.
49 *
50 * GR = Group Replication, a server-side plugin responsible for synchronising
51 * data between cluster nodes. It exposes certain dynamic tables (views)
52 * that we query to obtain health status of the cluster. It reflects the
53 * real "as it actually is" version of topology.
54 *
55 * MDC = MD Cache, this plugin (the code that you're reading now).
56 *
57 * MM = multi-primary, a replication mode in which all GR members are RW.
58 *
59 * SM = single-primary, a replication mode in which only one GR member is RW,
60 * rest are RO.
61 *
62 * ATTOW = contraction for "at the time of writing".
63 *
64 * [xx] (where x is a digit) = reference to a note in Notes section.
65 *
66 *
67 *
68 *
69 *
70 * Refresh Mechanism
71 * =================
72 *
73 *
74 *
75 * ## Overview
76 * MDC refresh runs in its own thread and periodically queries both MD and GR
77 * for status, then updates routing table which is queried by the Routing
78 * Plugin. Its entry point is `MetadataCache::start()`, which (indirectly) runs
79 * a "forever loop" in `MetadataCache::refresh_thread()`, which in turn is
80 * responsible for perodically running `MetadataCache::refresh()`.
81 *
82 * `MetadataCache::refresh()` is the "workhorse" of refresh mechanism.
83 *
84 *
85 *
86 *
87 *
88 * ## Refresh trigger
89 * `MetadataCache::refresh_thread()` call to `MetadataCache::refresh()` can be
90 * triggered in 2 ways:
91 * - `<TTL>` seconds passed since last refresh
92 * - X protocol notification triggered by the GR change in case of GR cluster
93 *
94 * It's implemented by running a sleep loop between refreshes. The loop sleeps 1
95 * second at a time, until `<TTL>` iterations have gone.
96 *
97 *
98 *
99 *
100 *
101 * ## Refresh process
102 * Once refresh is called, it goes through the following stages:
103 * - Stage 1: Query MD
104 * - Stage 2: Query GR, combine results with MD, determine availability
105 * - Stage 3: Update routing table
106 *
107 * In subsequent sections each stage is described in more detail
108 *
109 *
110 *
111 *
112 *
113 * ### Stage 1: Query MD
114 * This stage can be divided into two major substages:
115 * 1. Connect to MD server
116 * 2. Extract MD information
117 *
118 *
119 *
120 * #### Stage 1.1: Connect to MD server
121 *
122 * Implemented in: `ClusterMetadata::connect()`
123 *
124 * MDC starts with a list of MD servers written in the dynamic configuration
125 * (state) file, such as:
126 *
127 *
128 * "cluster-metadata-servers": [
129 * "mysql://192.168.56.101:3310",
130 * "mysql://192.168.56.101:3320",
131 * "mysql://192.168.56.101:3330"
132 * ]
133 *
134 * It iterates through the list and tries to connect to each one, until
135 * connection succeeds.
136 *
137 * @note
138 * This behavior might change in near future, because it does not ensure that
139 * connected MD server holds valid MD data [01].
140 *
141 * @note
142 * Iteration always starting from 1st server on the list might also change [02].
143 *
144 * @note
145 * New connection is always established and old one closed off, even if old one
146 * is still alive and usable.
147 *
148 *
149 *
150 * #### Stage 1.2: Extract MD Information
151 *
152 * Implemented in: `ClusterMetadata::fetch_instances_from_metadata_server()`
153 *
154 * Using connection established in Stage 1.1, MDC runs a SQL query which
155 * extracts a list of nodes (GR members) belonging to the cluster. Note that
156 * this the configured "should be" view of cluster topology, which might not
157 * correspond to actual topology, if for example some nodes became unavailable,
158 * changed their role or new nodes were added without updating MD in the
159 * server.
160 *
161 * @note
162 * ATTOW, if this query fails, whole refresh process fails [03].
163 *
164 *
165 *
166 *
167 *
168 * ### Stage 2: Query GR, combine results with MD, determine availability
169 *
170 * Implemented in: `ClusterMetadata::update_cluster_status()`
171 *
172 * Here MDC iterates through the list of GR members obtained from MD in Stage
173 * 1.2, until it finds a "trustworthy" GR node. A "trustworthy" GR node is one
174 * that passes the following substages:
175 * 1. successfully connects
176 * 2. successfully responds to 2 GR status SQL queries
177 * 3. is part of quorum (regardless of whether it's available or not)
178 *
179 * If MDC doesn't find a "trustworthy" node, it clears the routing table,
180 * resulting in Routing Plugin not routing any new connections.
181 *
182 * @note
183 * Since Stage 2 got its list of candidate GR nodes from MD server, it follows
184 * that MDC will never query any nodes not present in MD for GR status.
185 *
186 * @note
187 * Any routing table updates will not go into effect until Stage 3, where it is
188 * applied.
189 *
190 * @note
191 * ATTOW clearing routing table will not automatically close off old
192 * connections. This is a bug which is addressed by upcoming WL#11954.
193 *
194 *
195 *
196 * #### Stage 2.1: Connect to GR node
197 *
198 * Implemented in: `ClusterMetadata::update_cluster_status()`
199 *
200 * New connection to GR node is established (on failure, Stage 2 progresses to
201 * next iteration).
202 *
203 * @note
204 * Since connection to MD server in Stage 1.1 is not closed after that stage
205 * finishes, there's an optimisation done for when connecting to a GR member
206 * that's the same node as the MD server - in such case, the connection is
207 * simply re-used rather than new one opened.
208 *
209 *
210 *
211 * #### Stage 2.2: Extract GR status
212 *
213 * Implemented in: `fetch_group_replication_members()` and
214 * `find_group_replication_primary_member()`
215 *
216 * Two SQL queries are ran and combined to produce a status report of all nodes
217 * seen by this node (which would be the entire cluster if it was in perfect
218 * health, or its subset if some nodes became unavailable or the cluster was
219 * experiencing a split-brain scenario):
220 *
221 * 1. determine the PRIMARY member of the cluster (if there is more than
222 * one, such as in MM setups, the first one is returned and the rest are
223 * ignored)
224 *
225 * 2. get the membership and health status of all GR nodes, as seen by this
226 * node
227 *
228 * If either SQL query fails to execute, Stage 2 iterates to next GR node.
229 *
230 * @note
231 * ATTOW, 1st query is always ran, regardless of whether we're in MM mode or
232 * not. As all nodes are PRIMARY in MM setups, we could optimise this query away
233 * in MM setups.
234 *
235 *
236 *
237 * #### Stage 2.3: Quorum test
238 *
239 * Implemented in: `ClusterMetadata::update_cluster_status()` and
240 * `ClusterMetadata::check_cluster_status()`
241 *
242 * MD and GR data collected up to now are compared, to see if GR node just
243 * queried belongs to an available cluster (or to an available cluster
244 * partition, if cluster has partitioned). For a cluster (partition) to
245 * be considered available, it has to have quorum, that is, meet the following
246 * condition:
247 *
248 * count(ONLINE nodes) + count(RECOVERING nodes)
249 * is greater than
250 * 1/2 * count(all original nodes according to MD).
251 *
252 * If particular GR node does not meet the quorum condition, Stage 2 iterates
253 * to next GR node.
254 *
255 * OTOH, if GR node is part of quorum, Stage 2 will not iterate further,
256 * because it would be pointless (it's not possible to find a member that's
257 * part of another quorum, because there can only be one quorum, the one we
258 * just found). This matters, because having quorum does not automatically
259 * imply being available, as next paragraph explains.
260 *
261 * The availability test will resolve node's cluster to be in of the 4
262 * possible states:
263 * - Unavailable (this node is not part of quorum)
264 * - UnavailableRecovering (quorum is met, but it consists of only RECOVERING
265 * nodes - this is a rare cornercase)
266 * - AvailableWritable (quorum is met, at least one RW node present)
267 * - AvailableReadOnly (quorum is met, no RW nodes present)
268 *
269 * As already mentioned, reaching 1st of the 4 above states will result in
270 * Stage 2 iterating to the next node. OTOH, achieving one of remaining 3
271 * states will cause MDC to move on to Stage 3, where it sets the routing table
272 * accordingly.
273 *
274 *
275 * ##### GR-MD dishorenecy
276 *
277 * ATTOW, our Router has a certain limitation: it assumes that MD contains an
278 * exact set or superset of nodes in GR. The user is normally expected to use
279 * MySQL Shell to reconfigure the cluster, which automatically updates both
280 * GR and MD, keeping them in sync. But if for some reason the user tinkers with
281 * GR directly and adds nodes without updating MD accordingly,
282 * availablity/quorum calculations will be skewed. We run checks to detect such
283 * situation, and log a warning like so:
284 *
285 * log_error("Member %s:%d (%s) found in Group Replication, yet is not
286 * defined in metadata!"
287 *
288 * but beyond that we just act defensively by having our quorum calculation be
289 * conservative, and error on the side of caution when such discrepancy happens
290 * (quorum becomes harder to reach than if MD contained all GR members).
291 *
292 * Quorum is evaluated in code as follows:
293 *
294 * bool have_quorum = (quorum_count > member_status.size()/2);
295 *
296 * - `quorum_count` is the sum of PRIMARY, SECONDARY and RECOVERING nodes that
297 * appear in MD _and_ GR
298 * - `member_status.size()` is the sum of all nodes in GR, regardless of if they
299 * show up in MD or not
300 * - any nodes in MD but not in GR will be marked as Unavailable, this is an
301 * expected scenario, and we react correctly; they do not increment
302 * `quorum_count` nor `member_status.size()`
303 * - any nodes in GR but not in MD will never become routing destinations,
304 * but they will increment the `member_status.size()`, making quorum harder to
305 * reach.
306 *
307 * To illustrate how our quorum calculation will behave when GR and MD get out
308 * sync, below are some example scenarios:
309 *
310 * ###### Scenario 1
311 *
312 * MD defines nodes A, B, C
313 * GR defines nodes A, B, C, D, E
314 * A, B are alive; C, D, E are dead
315 *
316 * Availability calculation should deem cluster to be unavailable, because
317 * only 2 of 5 nodes are alive, even though looking purely from MD
318 * point-of-view, 2 of its 3 nodes are still alive, thus could be considered a
319 * quorum. In such case:
320 *
321 * quorum_count = 2 (A and B)
322 * member_status.size() = 5
323 *
324 * and thus:
325 *
326 * have_quorum = (2 > 5/2) = false
327 *
328 * ###### Scenario 2
329 *
330 * MD defines nodes A, B, C
331 * GR defines nodes A, B, C, D, E
332 * A, B are dead, C, D, E are alive
333 *
334 * Availability calculation, if fully GR-aware, could deem cluster as
335 * available, because looking from purely GR perspective, 3 of 5 nodes form
336 * quorum. OTOH, looking from MD perspective, only 1 of 3 its nodes (C) is
337 * alive.
338 *
339 * Our availability calculation prefers to err on the side of caution. So here
340 * the availability is judged as not available, even though it could be. But
341 * that's the price we pay in exchange for the safety the algorithm provides
342 * demonstrated in the previous scenario:
343 *
344 * quorum_count = 1 (C)
345 * member_status.size() = 5
346 *
347 * and thus:
348 *
349 * have_quorum = (1 > 5/2) = false
350 *
351 * ###### Scenario 3
352 *
353 * MD defines nodes A, B, C
354 * GR defines nodes C, D, E
355 * A, B are not reported by GR, C, D, E are alive
356 *
357 * According to GR, there's a quorum between nodes C, D and E. However, from MD
358 * point-of-view, A and B went missing and only C is known to be alive.
359 *
360 * Again, our available calculation prefers to err on the safe side:
361 *
362 * quorum_count = 1 (C)
363 * member_status.size() = 5
364 *
365 * and thus:
366 *
367 * have_quorum = (1 > 5/2) = false
368 *
369 *
370 * ##### Why don't we just use GR data (and do away with Metadata)?
371 *
372 * Need for cluster configuration aside, there is another challenge. GR can
373 * provide IP/hostnames of nodes as it sees them from its own perspective, but
374 * those IP/hostnames might not be externally-reachable. OTOH, MD tables provide
375 * external IP/hostnames which Router relies upon to reach the GR nodes.
376 *
377 *
378 *
379 *
380 *
381 * ### Stage 3: Update routing table
382 *
383 * Implemented in: `MetadataCache::refresh()`
384 *
385 * Once stage 2 is complete, the resulting routing table from Stage 2 is
386 * applied. It is also compared to the old routing table and if there is a
387 * difference between them then appropriate log messages are issued advising of
388 * availability change.
389 *
390 *
391 *
392 *
393 *
394 * ### Stage 1.1
395 * [01] There has been a recent concern ATTOW, that MD returned might be stale,
396 * if MD server node is in RECOVERING state. This assumes the MD server is
397 * also deployed on an InnoDB cluster.
398 *
399 * [02] It might be better to always start from the last successfully-connected
400 * server, rather than 1st on the list, to avoid unneccessary connection
401 * attempts when 1st server is dead.
402 *
403 *
404 * ### Stage 1.2
405 * [03] If MD-fetching SQL statement fails to execute or process properly, it
406 * will raise an exception that is caught by the topmost catch of the
407 * refresh process, meaning, another MD server will not be queried. This is
408 * a bug, reported as BUG#28082473
409 */
