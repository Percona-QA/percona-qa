170 /*
171 Recovery core method:
172
173 * Step 0: Declare recovery running after extracting group information
174
175 * Step 1: Wait for the applier to execute pending transactions and suspend.
176 Even if the joiner is alone, it goes trough this phase so it is declared
177 ONLINE only when it executed all pending local transactions.
178
179 * Step 2: Declare the node ONLINE if alone.
180 This is done solely based on the number of member the group had when
181 recovery started. No further group changes affect this decision.
182
183 * Step 3: State transfer.
184 This can be summarized as:
185 1) Connect to a donor
186 2) Wait until the data comes.
187 It can be interrupted/terminated by:
188 > recovery_aborted is set to true. This means recovery was aborted.
189 The wait is awaken and the loop is broken. The thread shutdowns.
190 > on_failover is set to true. This means the current donor left.
191 The loop cycles and another donor is selected.
192 The threads are stopped but the logs are not purged.
193 A connections is established to another donor.
194 > donor_channel_applier_error is set to true. This means an error was
195 detected in the recovery applier.
196 When the loop cycles, kill the threads and purge the logs
197 A connections is established to another donor.
198 > donor_transfer_finished. This means we received all the data.
199 The loop exits
200
201 * Step 4: Awake the applier and wait for the execution of cached transactions.
202
203 * Step 5: Notify the group that we are now online if no error occurred.
204 This is done even if the member is alone in the group.
205
206 * Step 6: If an error occurred and recovery is impossible leave the group.
207 We leave the group but the plugin is left running.
208
209 * Step 7: Terminate the recovery thread.
210 */
