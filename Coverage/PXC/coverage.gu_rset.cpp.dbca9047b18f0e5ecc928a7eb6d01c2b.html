133 * Since in VER2 we want everything to be aligned to 8 bytes, we can have
134 * an important optimization for smaller sets by fitting count and size
135 * into bytes 1-3 of the header, thus fitting whole header into 8 bytes:
136 *
137 * | BYTE 0 | BYTE 1 | BYTE 2 | BYTE 3 | 4 bytes
138 * | VERSION BYTE | COUNT BITS | SIZE BITS | header checksum
139 *
140 * Optimizing for maximum count of 16-byte records, so size = count*16.
141 * This will allow to encode up to 1K records of total 16K size. This is
142 * more than can be represented by VLQ in 1 and 2 bytes respectively:
143 * 127 records of total 16K-1 size.
