567 /**
568 While there are more unprocessed rows ready to process given the current

570 the window functions defined over this window on the current frame, moving
571 the frame if required.
572
573 This method contains the main execution time logic of the evaluation
574 window functions if we need buffering for one or more of the window functions
575 defined on the window.
576
577 Moving (sliding) frames can be executed using a naive or optimized strategy
578 for aggregate window functions, like SUM or AVG (but not MAX, or MIN).
579 In the naive approach, for each row considered for processing from the buffer,
580 we visit all the rows defined in the frame for that row, essentially leading
581 to N*M complexity, where N is the number of rows in the result set, and M is
582 the number for rows in the frame. This can be slow for large frames,
583 obviously, so we can choose an optimized evaluation strategy using inversion.
584 This means that when rows leave the frame as we move it forward, we re-use
585 the previous aggregate state, but compute the *inverse* function to eliminate
586 the contribution to the aggregate by the row(s) leaving the frame, and then
587 use the normal aggregate function to add the contribution of the rows moving
588 into the frame. The present method contains code paths for both strategies.
589

591 same if no overflow occurs during normal evaluation. For floating numbers,
592 optimizing in this way may lead to different results, so it is not done by
593 default, cf the session variable "windowing_use_high_precision".
