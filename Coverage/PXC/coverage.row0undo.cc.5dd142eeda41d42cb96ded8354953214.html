53 /* How to undo row operations?
54 (1) For an insert, we have stored a prefix of the clustered index record
55 in the undo log. Using it, we look for the clustered record, and using
56 that we look for the records in the secondary indexes. The insert operation
57 may have been left incomplete, if the database crashed, for example.
58 We may have look at the trx id and roll ptr to make sure the record in the
59 clustered index is really the one for which the undo log record was
60 written. We can use the framework we get from the original insert op.
61 (2) Delete marking: We can use the framework we get from the original
62 delete mark op. We only have to check the trx id.
63 (3) Update: This may be the most complicated. We have to use the framework
64 we get from the original update op.
65
66 What if the same trx repeatedly deletes and inserts an identical row.
67 Then the row id changes and also roll ptr. What if the row id was not
68 part of the ordering fields in the clustered index? Maybe we have to write
69 it to undo log. Well, maybe not, because if we order the row id and trx id
70 in descending order, then the only undeleted copy is the first in the
71 index. Our searches in row operations always position the cursor before
72 the first record in the result set. But, if there is no key defined for
73 a table, then it would be desirable that row id is in ascending order.
74 So, lets store row id in descending order only if it is not an ordering
75 field in the clustered index.
76
77 NOTE: Deletes and inserts may lead to situation where there are identical
78 records in a secondary index. Is that a problem in the B-tree? Yes.
79 Also updates can lead to this, unless trx id and roll ptr are included in
80 ord fields.
81 (1) Fix in clustered indexes: include row id, trx id, and roll ptr
82 in node pointers of B-tree.
83 (2) Fix in secondary indexes: include all fields in node pointers, and
84 if an entry is inserted, check if it is equal to the right neighbor,
85 in which case update the right neighbor: the neighbor must be delete
86 marked, set it unmarked and write the trx id of the current transaction.
87
88 What if the same trx repeatedly updates the same row, updating a secondary
89 index field or not? Updating a clustered index ordering field?
90
91 (1) If it does not update the secondary index and not the clustered index
92 ord field. Then the secondary index record stays unchanged, but the
93 trx id in the secondary index record may be smaller than in the clustered
94 index record. This is no problem?
95 (2) If it updates secondary index ord field but not clustered: then in
96 secondary index there are delete marked records, which differ in an
97 ord field. No problem.
98 (3) Updates clustered ord field but not secondary, and secondary index
99 is unique. Then the record in secondary index is just updated at the
100 clustered ord field.
101 (4)
102
103 Problem with duplicate records:
104 Fix 1: Add a trx op no field to all indexes. A problem: if a trx with a
105 bigger trx id has inserted and delete marked a similar row, our trx inserts
106 again a similar row, and a trx with an even bigger id delete marks it. Then
107 the position of the row should change in the index if the trx id affects
108 the alphabetical ordering.
109
110 Fix 2: If an insert encounters a similar row marked deleted, we turn the
111 insert into an 'update' of the row marked deleted. Then we must write undo
112 info on the update. A problem: what if a purge operation tries to remove
113 the delete marked row?
114
115 We can think of the database row versions as a linked list which starts
116 from the record in the clustered index, and is linked by roll ptrs
117 through undo logs. The secondary index records are references which tell
118 what kinds of records can be found in this linked list for a record
119 in the clustered index.
120
121 How to do the purge? A record can be removed from the clustered index
122 if its linked list becomes empty, i.e., the row has been marked deleted
123 and its roll ptr points to the record in the undo log we are going through,
124 doing the purge. Similarly, during a rollback, a record can be removed
125 if the stored roll ptr in the undo log points to a trx already (being) purged,
126 or if the roll ptr is NULL, i.e., it was a fresh insert. */
