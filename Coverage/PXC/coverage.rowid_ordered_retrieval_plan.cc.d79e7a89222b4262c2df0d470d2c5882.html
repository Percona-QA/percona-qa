364 Get selectivity of adding a ROR scan to the ROR-intersection.
365
366 SYNOPSIS
367 ror_scan_selectivity()
368 info ROR-interection, an intersection of ROR index scans
369 scan ROR scan that may or may not improve the selectivity
370 of 'info'
371
372 NOTES
373 Suppose we have conditions on several keys
374 cond=k_11=c_11 AND k_12=c_12 AND ... // key_parts of first key in 'info'
375 k_21=c_21 AND k_22=c_22 AND ... // key_parts of second key in 'info'
376 ...
377 k_n1=c_n1 AND k_n3=c_n3 AND ... (1) //key_parts of 'scan'
378
379 where k_ij may be the same as any k_pq (i.e. keys may have common parts).
380
381 Note that for ROR retrieval, only equality conditions are usable so there
382 are no open ranges (e.g., k_ij > c_ij) in 'scan' or 'info'.
383 FIXME: This isn't true in practice; e.g. i_main.costmodel_planchange ends
384 up calling this function with an inequality condition, and thus the
385 estimation is probably wrong (since the code assumes only one element in the
386 tree).
387
388 A full row is retrieved if entire condition holds.
389
390 The recursive procedure for finding P(cond) is as follows:
391
392 First step:
393 Pick 1st part of 1st key and break conjunction (1) into two parts:
394 cond= (k_11=c_11 AND R)
395
396 Here R may still contain condition(s) equivalent to k_11=c_11.
397 Nevertheless, the following holds:
398
399 P(k_11=c_11 AND R) = P(k_11=c_11) * P(R | k_11=c_11).
400
401 Mark k_11 as fixed field (and satisfied condition) F, save P(F),
402 save R to be cond and proceed to recursion step.
403
404 Recursion step:
405 We have a set of fixed fields/satisfied conditions) F, probability P(F),
406 and remaining conjunction R
407 Pick next key part on current key and its condition "k_ij=c_ij".
408 We will add "k_ij=c_ij" into F and update P(F).
409 Lets denote k_ij as t, R = t AND R1, where R1 may still contain t. Then
410
411 P((t AND R1)|F) = P(t|F) * P(R1|t|F) = P(t|F) * P(R1|(t AND F)) (2)
412
413 (where '|' mean conditional probability, not "or")
414
415 Consider the first multiplier in (2). One of the following holds:
416 a) F contains condition on field used in t (i.e. t AND F = F).
417 Then P(t|F) = 1
418
419 b) F doesn't contain condition on field used in t. Then F and t are
420 considered independent.
421
422 P(t|F) = P(t|(fields_before_t_in_key AND other_fields)) =
423 = P(t|fields_before_t_in_key).
424
425 P(t|fields_before_t_in_key) = #records(fields_before_t_in_key) /
426 #records(fields_before_t_in_key, t)
427
428 The second multiplier is calculated by applying this step recursively.
429
430 IMPLEMENTATION
431 This function calculates the result of application of the "recursion step"
432 described above for all fixed key members of a single key, accumulating set
433 of covered fields, selectivity, etc.
434
435 The calculation is conducted as follows:
436 Lets denote #records(keypart1, ... keypartK) as n_k. We need to calculate
437
438 n_{k1} n_{k2}
439 --------- * --------- * .... (3)
440 n_{k1-1} n_{k2-1}
441

444 parts of a single key).
445 Since it is reasonable to expect that most of the fields are not marked

448 n_{i1} n_{i2}
449 (3) = n_{max_key_part} / ( --------- * --------- * .... )
