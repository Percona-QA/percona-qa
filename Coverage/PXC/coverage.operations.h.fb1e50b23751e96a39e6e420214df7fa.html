186 @page PAGE_COMPONENT_KEYRING_WRITE_NEW "How to write a new keyring component"
187
188 Common keyring implementation infrastructure provides useful parts to minimize
189 efforts involved in writing a new component. This page will provide details about
190 how a new keyring component can be implemented using the common infrastructure
191
192 Any keyring component can be divided in 3 parts:
193
194 1. Configuration Management
195 1. Service Implementation
196 3. Backend Management
197
198 @section configuration_management Configuration Management
199 A binary must be able to load a keyring component on top of minimal chassis.
200 Hence, the component cannot use option management provided by underlying binary
201 to read configuration for the keyring.
202
203 One way to read configuration is to accept it as a JSON file.
204 A sample configuration file may look like following:
205 @code
206 {
207 "version" : "1.0.0"
208 "config_key_1" : "config_data_1",
209 "config_key_2" : "config_data_2",
210 "config_key_3" : "config_data_3"
211 }
212 @endcode
213
214 Various (key, value) pair would help component get details required
215 to communicate with backend and initialize keyring.
216
217 You can use @ref keyring_common::config::Config_reader
218 to read a JSON file and extract value of individual configuration parameters.
219
220 Depending upon the services implemented by the component, it is possible that:
221
222 A> Configuration data can be queried at runtime
223
224 B> Configuration can be changed at runtime
225
