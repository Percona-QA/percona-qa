142 //
143 // About transaction states:
144 //
145 // The TrxHandleMaster stats are used to track the state of the
146 // transaction, while TrxHandleSlave states are used to track
147 // which critical sections have been accessed during write set
148 // applying. As a convention, TrxHandleMaster states are changed
149 // before entering the critical section, TrxHandleSlave states
150 // after critical section has been successfully entered.
151 //
152 // TrxHandleMaster states during normal execution:
153 //
154 // EXECUTING - Transaction handle has been created by appending key
155 // or write set data
156 // REPLICATING - Transaction write set has been send to group
157 // communication layer for ordering
158 // CERTIFYING - Transaction write set has been received from group
159 // communication layer, has entered local monitor and
160 // is certifying
161 // APPLYING - Transaction has entered applying critical section
162 // COMMITTING - Transaction has entered committing critical section
163 // COMMITTED - Transaction has released commit time critical section
164 // ROLLED_BACK - Application performed a voluntary rollback
165 //
166 // Note that streaming replication rollback happens by replicating
167 // special rollback writeset which will go through regular write set
168 // critical sections.
169 //
170 // Note/Fixme: CERTIFYING, APPLYING and COMMITTING states seem to be
171 // redundant as these states can be tracked via
172 // associated TrxHandleSlave states.
173 //
174 //
175 // TrxHandleMaster states after effective BF abort:
176 //
177 // MUST_ABORT - Transaction enter this state after successful BF abort.
178 // BF abort is allowed if:
179 // * Transaction does not have associated TrxHandleSlave
180 // * Transaction has associated TrxHandleSlave but it does
181 // not have commit flag set
182 // * Transaction has associated TrxHandleSlave, commit flag
183 // is set and the TrxHandleSlave global sequence number is
184 // higher than BF aborter global sequence number
185 //
186 // 1) If the certification after BF abort results a failure:
187 // ABORTING - BF abort was effective and certification
188 // resulted a failure
189 // ROLLING_BACK - Commit order critical section has been grabbed for
190 // rollback
191 // ROLLED_BACK - Commit order critical section has been released after
192 // successful rollback
193 //
194 // 2) The case where BF abort happens after successful certification or
195 // if out-of-order certification results a success:
196 // MUST_REPLAY - The transaction must roll back and replay in applier
197 // context.
198 // * If the BF abort happened before certification,
199 // certification must be performed in applier context
200 // and the transaction replay must be aborted if
201 // the certification fails.
202 // * TrxHandleSlave state can be used to determine
203 // which critical sections must be entered before the
204 // replay. For example, if the TrxHandleSlave state is
205 // REPLICATING, write set must be certified under local
206 // monitor and both apply and commit monitors must be
207 // entered before applying. On the other hand, if
208 // TrxHandleSlave state is APPLYING, only commit monitor
209 // must be grabbed before replay.
210 //
211 // TrxHandleMaster states after replication failure:
212 //
213 // ABORTING - Replicaition resulted a failure
214 // ROLLING_BACK - Error has been returned to application
215 // ROLLED_BACK - Application has finished rollback
216 //
217 //
218 // TrxHandleMaster states after certification failure:
219 //
220 // ABORTING - Certification resulted a failure
221 // ROLLING_BACK - Commit order critical section has been grabbed for
222 // rollback
223 // ROLLED_BACK - Commit order critical section has been released
224 // after successful rollback
225 //
226 //
227 //
228 // TrxHandleSlave:
229 // REPLICATING - this is the first state for TrxHandleSlave after it
230 // has been received from group
231 // CERTIFYING - local monitor has been entered successfully
232 // APPLYING - apply monitor has been entered successfully
233 // COMMITTING - commit monitor has been entered successfully
234 //
235 // TrxHandleSlave state machine is restricted in order to use it
236 // for tracking which monitors have been entered. Certification result
237 // can be queried via is_dummy().
238 //
239 // State machine diagrams can be found below.
