488 Do the final touchups of the access path tree, once we have selected a final
489 plan (ie., there are no more alternatives). There are currently two major
490 tasks to do here: Account for materializations (because we cannot do it until
491 we have the entire plan), and set up filesorts (because it involves
492 constructing new objects, so we don't want to do it for unused candidates).
493 The former also influences the latter.
494
495 Materializations in particular are a bit tricky due to the way our item system
496 works; expression evaluation cares intimately about _where_ values come from,
497 not just what they are (i.e., all non-leaf Items carry references to other
498 Items, and pull data only from there). Thus, whenever an Item is materialized,
499 references to that Item need to be modified to instead point into the correct
500 field in the temporary table. We traverse the tree bottom-up and keep track of
501 which materializations are active, and modify the appropriate Item lists at
502 any given point, so that they point to the right place. We currently modify:
503
504 - The SELECT list. (There is only one, so we can update it as we go.)
505 - Referenced fields for INSERT ... ON DUPLICATE KEY UPDATE (IODKU);
506 also updated as we go.
507 - Sort keys (e.g. for ORDER BY).
508 - The HAVING clause, if the materialize node is below an aggregate node.
509 (If the materialization is above aggregation, the HAVING clause has
510 already accomplished its mission of filtering out the uninteresting
511 results, and will not be evaluated anymore.)
512
513 Surprisingly enough, we also need to update the materialization parameters
514 themselves. Say that we first have a materialization that copies
515 t1.x -> <temp1>.x. After that, we have a materialization that copies
516 t1.x -> <temp2>.x. For this to work properly, we obviously need to go in
517 and modify the second one so that it instead says <temp1>.x -> <temp2>.x,
518 ie., the copy is done from the correct source.
519
520 You cannot yet insert temporary tables in arbitrary places in the query;
521 in particular, we do not yet handle these rewrites (although they would
522 very likely be possible):
523
524 - Group elements for aggregations (GROUP BY). Do note that
525 create_tmp_table() will replace elements within aggregate functions
526 if you set save_sum_funcs=false; you may also want to supplant
527 this mechanism.
528 - Filters (e.g. WHERE predicates); do note that partial pushdown may
