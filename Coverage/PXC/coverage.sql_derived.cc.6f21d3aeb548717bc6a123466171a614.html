70 TABLE_LIST 'tl', to have a single materialization of multiple references to

73 How sharing of a single tmp table works
74 =======================================
75
76 There are several scenarios.
77 (1) Non-recursive CTE referenced only once: nothing special.
78 (2) Non-recursive CTE referenced more than once:
79 - multiple TABLEs, one TABLE_SHARE.
80 - The first ref in setup_materialized_derived() calls
81 create_tmp_table(); others call open_table_from_share().
82 - The first ref in create_derived() calls instantiate_tmp_table()
83 (which calls handler::create() then open_tmp_table()); others call
84 open_tmp_table(). open_tmp_table() calls handler::open().
85 - The first ref in materialize_derived() evaluates the subquery and does
86 all writes to the tmp table.
87 - Finally all refs set up a read access method (table scan, index scan,
88 index lookup, etc) and do reads, possibly interlaced (example: a
89 nested-loop join of two references to the CTE).
90 - The storage engine (MEMORY or InnoDB) must be informed of the uses above;
91 this is done by having TABLE_SHARE::ref_count>=2 for every handler::open()
92 call.
93 (3) Recursive CTE, referenced once or more than once:
94 All of (2) applies, where the set of refs is the non-recursive
95 ones (a recursive ref is a ref appearing in the definition of a recursive
96 CTE). Additionally:
97 - recursive refs do not call setup_materialized_derived(),
98 create_derived(), materialize_derived().
99 - right after a non-recursive ref has been in setup_materialized_derived(),
100 its recursive refs are replaced with clones of that ref, made with
101 open_table_from_share().
102 - the first non-recursive ref in materialized_derived() initiates the
103 with-recursive algorithm:
104 * its recursive refs call open_tmp_table().
105 * Then writes (to the non-recursive ref) and reads (from the recursive
106 refs) happen interlaced.
107 - a particular recursive ref is the UNION table, if UNION DISTINCT is
108 present in the CTE's definition: there is a single TABLE for it,
109 writes/reads to/from it happen interlaced (writes are done by
110 Query_result_union::send_data(); reads are done by the fake_query_block's
111 JOIN).
112 - Finally all non-recursive refs set up a read access method and do reads,
113 possibly interlaced.
114 - The storage engine (MEMORY or InnoDB) must be informed of the uses above;
115 this is done by having TABLE_SHARE::ref_count>=2 for every handler::open()
116 call.
117 - The Server code handling tmp table creation must also be informed:
118 see how Query_result_union::create_result_table() disables PK promotion.
119
120 How InnoDB manages the uses above
121 =================================
122
123 The storage engine needs to take measures so that inserts and reads
124 don't corrupt each other's behaviour. In InnoDB that means two things
125 (@see row_search_no_mvcc()):
126 (a) A certain way to use its cursor when reading
127 (b) Making the different handlers inform each other when one insertion
128 modifies the structure of the index tree (e.g. splits a page; this
129 triggers a refreshing of all read cursors).
130
131 Requirements on tmp tables used to write/read CTEs
132 ==================================================
133
134 The internal tmp table must support a phase where table scans and
135 insertions happen interlaced, either issued from a single TABLE or from
136 multiple TABLE clones. If from a single TABLE, that object does repetitions
137 of {"write rows" then "init scan / read rows / close scan"}. If from
138 multiple TABLEs, one does "write rows", every other one does "init scan /
139 read rows / close scan".
140 During this, neither updates, nor deletes, nor any other type of read
141 access than table scans, are allowed on this table (they are allowed after
142 the phase's end).
143 Any started table scan on this table:
144 - must remember its position between two read operations, without influence
145 from other scans/inserts;
146 - must return rows inserted before and after it started (be catching up
147 continuously) (however, when it reports EOF it is allowed to stop catching
148 up and report EOF until closed).
149 - must return rows in insertion order.
150 - may be started from the first record (ha_rnd_init, ha_rnd_next) or from
151 the record where the previous scan was ended (position(), ha_rnd_end,
152 [...], ha_rnd_init, ha_rnd_pos(saved position), ha_rnd_next).
153 - must return positions (handler::position()) which are stable if a write
154 later occurs, so that a handler::rnd_pos() happening after the write finds
155 the same record.
156
157 Cursor re-positioning when MEMORY is converted to InnoDB
158 ========================================================

160 See create_ondisk_from_heap(). A requirement is that InnoDB is able to
161 start a scan like this: rnd_init, rnd_pos(some PK value), rnd_next.
