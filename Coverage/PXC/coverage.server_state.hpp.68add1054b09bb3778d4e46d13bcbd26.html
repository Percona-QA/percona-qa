20 /** @file server_state.hpp
21 *
22 * Server State Abstraction
23 * ==========================
24 *
25 * This file defines an interface for WSREP Server State.
26 * The Server State will encapsulate server identification,
27 * server state and server capabilities. The class also
28 * defines an interface for manipulating server state, applying
29 * of remote transaction write sets, processing SST requests,
30 * creating local client connections for local storage access
31 * operations.
32 *
33 * Concepts
34 * ========
35 *
36 * State Snapshot Transfer
37 * -----------------------
38 *
39 * Depending on SST type (physical or logical), the server storage
40 * engine initialization must be done before or after SST happens.
41 * In case of physical SST method (typically rsync, filesystem snapshot)
42 * the SST happens before the storage engine is initialized, in case
43 * of logical backup typically after the storage engine initialization.
44 *
45 * Rollback Mode
46 * -------------
47 *
48 * When High Priority Transaction (HPT) write set is applied, it
49 * may be required that the HPT Brute Force Aborts (BFA) locally
50 * executing transaction. As HPT must be able to apply all its
51 * write sets without interruption, the locally executing transaction
52 * must yield immediately, otherwise a transaction processing
53 * may stop or even deadlock. Depending on DBMS implementation,
54 * the local transaction may need to be rolled back immediately
55 * (synchronous mode) or the rollback may happen later on
56 * (asynchronous mode). The Server Context implementation
57 * which derives from Server Context base class must provide
58 * the base class for the rollback mode which server operates on.
59 *
60 * ### Synchronous
61 *
62 * If the DBMS server implementation does not allow asynchronous rollback,
63 * the victim transaction must be rolled back immediately in order to
64 * allow transaction processing to proceed. Depending on DBMS process model,
65 * there may be either background thread which processes the rollback
66 * or the rollback can be done by the HTP applier.
67 *
68 * ### Asynchronous
69 *
70 * In asynchronous mode the BFA victim transaction is just marked
71 * to be aborted or in case of fully optimistic concurrency control,
72 * the conflict is detected at commit.
73 *
74 *
75 * # Return value conventions
76 *
77 * The calls which are proxies to corresponding provider functionality
78 * will return wsrep::provider::status enum as a result. Otherwise
79 * the return value is generally zero on success, non zero on failure.
80 */
