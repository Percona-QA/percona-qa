56 /*
57 PartitionPruningModule
58
59 This part of the code does partition pruning. Partition pruning solves the
60 following problem: given a query over partitioned tables, find partitions
61 that we will not need to access (i.e. partitions that we can assume to be
62 empty) when executing the query.
63 The set of partitions to prune doesn't depend on which query execution
64 plan will be used to execute the query.
65
66 HOW IT WORKS
67
68 Partition pruning module makes use of RangeAnalysisModule. The following
69 examples show how the problem of partition pruning can be reduced to the
70 range analysis problem:
71
72 EXAMPLE 1
73 Consider a query:
74
75 SELECT * FROM t1 WHERE (t1.a < 5 OR t1.a = 10) AND t1.a > 3 AND t1.b='z'
76
77 where table t1 is partitioned using PARTITION BY RANGE(t1.a). An apparent
78 way to find the used (i.e. not pruned away) partitions is as follows:
79
80 1. analyze the WHERE clause and extract the list of intervals over t1.a
81 for the above query we will get this list: {(3 < t1.a < 5), (t1.a=10)}
82
83 2. for each interval I
84 {
85 find partitions that have non-empty intersection with I;
86 mark them as used;
87 }
88
89 EXAMPLE 2
90 Suppose the table is partitioned by HASH(part_func(t1.a, t1.b)). Then
91 we need to:
92
93 1. Analyze the WHERE clause and get a list of intervals over (t1.a, t1.b).
94 The list of intervals we'll obtain will look like this:
95 ((t1.a, t1.b) = (1,'foo')),
96 ((t1.a, t1.b) = (2,'bar')),
97 ((t1,a, t1.b) > (10,'zz'))
98
99 2. for each interval I
100 {
101 if (the interval has form "(t1.a, t1.b) = (const1, const2)" )
102 {
103 calculate HASH(part_func(t1.a, t1.b));
104 find which partition has records with this hash value and mark
105 it as used;
106 }
107 else
108 {
109 mark all partitions as used;
110 break;
111 }
112 }
113
114 For both examples the step #1 is exactly what RangeAnalysisModule could
115 be used to do, if it was provided with appropriate index description
116 (array of KEY_PART structures).
117 In example #1, we need to provide it with description of index(t1.a),
118 in example #2, we need to provide it with description of index(t1.a, t1.b).
119
120 These index descriptions are further called "partitioning index
121 descriptions". Note that it doesn't matter if such indexes really exist,
122 as range analysis module only uses the description.
123
124 Putting it all together, partitioning module works as follows:
125
126 prune_partitions() {
127 call create_partition_index_description();
128
129 call get_mm_tree(); // invoke the RangeAnalysisModule
130
131 // analyze the obtained interval list and get used partitions
132 call find_used_partitions();
133 }
134

618 We can find out which [sub]partitions to use if we obtain restrictions on

638 | ... ... $
639 | | | $
640 | +---------+ +---------+ $ +-----------+ +-----------+

655 +---------+ $ +------------+ +------------+

670 For the example in the above picture, one of stack states is:
671 in find_used_partitions(key_tree = "subpar2=c5") (***)
672 in find_used_partitions(key_tree = "subpar1=c3")
673 in find_used_partitions(key_tree = "par2=c2") (**)
674 in find_used_partitions(key_tree = "par1=c1")

677 depth (**), we find which partition(s) correspond to "par1=c1 AND par2=c2",

679 When we reach the depth (***), we find which subpartition(s) correspond to

681 appropriate subpartitions as used.
682
683 It is possible that constraints on some partitioning fields are missing.
684 For the above example, consider this stack state:
